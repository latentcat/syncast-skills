# Syncast 平台与创作工作流指南

这份指南面向“像人类一样操作 Syncast 项目”的外部 Agent。它不是 action API 参考，而是帮助 Agent 理解平台、项目结构和创作思路的上手说明。

## 平台心智模型

Syncast 是一个本地优先、多人协作的 AI 创作工作台，用来在同一个项目里组织：

- 文档：项目规范、剧本、角色设定、分镜说明、提示词规范。
- 资源：图片、视频、音频、参考素材、生成结果和文件夹结构。
- 画布：视觉参考、分镜结构、素材关系和创作草图。
- 时间轴：视频素材的粗剪、排序、预览和后续导出。
- AI / Agent：Imagine 生成图片/视频/音频，以及内部 Agent 协作处理复杂业务任务。
- 工作流：把重复创作步骤编排成节点式自动化流程。

外部 Agent 不应把 Syncast 当作单个聊天窗口。更合适的理解是：Syncast 是项目创作操作系统；外部 Agent 是项目操作者，内部 Agent 是平台内的业务执行者。

## 外部 Agent 的角色

外部 Agent 的目标不是自己绕过 Syncast 写内容，而是像人类用户一样：

1. 先理解项目现场。
2. 再判断用户目标属于规范、素材、画布、时间轴、生成、工作流还是内部 Agent 协作。
3. 对需要创意构思、业务判断或未知结构的任务，委托 Syncast 内部 Agent 返回草案或判断。
4. 对用户已给出完整内容或精确修改目标的任务，直接使用项目 GraphQL 写入并回读验收。
5. 等待任务完成后读取文档、资源、频道消息或任务结果。
6. 向用户汇报结果、缺口和下一步需要确认的操作。

如果用户说“帮我操作这个项目”“继续创作”“整理一下项目”，不要一上来就发 Agent 消息或生成图片。应先巡检项目，再选择行动。

## 产物受众与内容隔离

本指南本身是外部 Agent 操作指南，可以讲 CLI、Bridge、Action 和 GraphQL；这些接入知识不是项目创作内容。写入项目前必须先把目标分为三类：

- 外部 Agent 操作/集成指南：可以说明如何连接和调用项目，但除非用户明确要求集成文档，否则不写入项目内部。
- 项目内部 Skill、Agent instructions、项目规范和提示词模板：只描述业务规则、创作能力语义、模型 ID 与参数。使用“项目内 Imagine 能力”这类表达，不写 CLI、Bridge、standalone API、`syncast.*` Action、GraphQL mutation 名、传输或接入方式。
- 用户阅读文档：用产品语言和创作语言说明目标、规则、素材与产物，不暴露接入实现；明确的开发/集成文档除外。

隔离检查覆盖标题/名称、描述、正文/说明、提示词模板字段和元数据。不能出现“正文清理了，但 description 里仍有 CLI”的情况。产品能力名（Imagine、Assets、Docs、Canvas、时间轴、项目 Agent）、模型 ID、创作参数和真实项目引用可以保留。

示例：项目内部 Skill 应写“使用项目内 Imagine 能力，模型为 `oai-gpt-image-2`，比例为 16:9，分辨率为 2K”；不得写“调用 `syncast.imagine.submit`，不能调用 standalone CLI”。

## 接手项目时的第一判断

外部 Agent 每次加入项目后，第一步都是检查用户当前打开的项目状态，并判断它属于哪一种：

- 刚初始化后的空项目：缺少文档、资源、时间轴、明确创作规范。用户尚未提供结构时先讨论项目类型并委托内部 Agent 做创作草案；用户已经给出完整规范正文时直接写入，不再委托改写。
- 已经进行过操作的项目：已有文档、资源、画布、时间轴、频道消息或运行历史，先读取已有内容，避免重复创建和重复生成。

接着根据用户初始输入或直接询问用户，判断操作模式：

- 协作创作：外部 Agent 高频和用户沟通，提出选项、确认成本、让用户决定文档形式、时间轴 Slot、关键帧等创作路径。
- 完全托管：用户授权外部 Agent 在约定目标和预算内主动推进。即使托管，也要保留成本意识、引用校验和阶段性汇报，不要无计划地批量生成。

如果用户没有明确选择模式，默认按“协作创作”处理。

## 推荐创作路线

典型 AI 影像 / 短剧 / 创意项目可以按下面顺序推进：

1. 建立项目目标：确认项目名称、简介、风格、交付物和限制。
2. 建立项目规范：在文档中整理剧本、角色设定、世界观、场景、镜头语言和提示词规范。
3. 准备素材资源：上传或整理参考图、角色图、场景图、视频、音频和历史生成结果。
4. 组织视觉结构：用画布整理参考、分镜、镜头关系和视觉方向。
5. 发起生成：用 Imagine 或内部 Agent 生成图片、视频、音乐、镜头或补充素材。
6. 整理产物：把生成结果放回资源、文档、画布或时间轴。
7. 时间轴粗剪：把可用素材组织成视频结构。
8. 复查与迭代：读取任务结果，检查是否符合项目规范，再决定下一轮生成或编辑。

外部 Agent 每次接手时都要判断项目处于哪一步，而不是机械执行用户的一句话。

## Tab 与 Channel 规划

Syncast 的各个 tab 对应用户在不同阶段习惯看到的操作方式：

- 画布：组织创意、视觉参考、分镜和素材关系。
- 资源：整理文件夹、参考资产、生成结果和可复用素材。
- 文档：编写项目规范、剧本、角色、场景、镜头说明和提示词规范。
- 时间轴：做粗剪、镜头想法排布、AI 待生成 Slot 和关键帧规划。

每个 tab 右侧都可以展开 Imagine Panel 或 Agent Panel；其中每个 Channel 用户都能看到。因此外部 Agent 推送任务时要规划 Channel：

- 同一主题、同一阶段的连续任务应复用同一个 Channel。
- 不要为每个任务都新建 Channel，否则用户很难回看。
- 可按用途建立少量稳定 Channel，例如“项目规划 / 角色设定 / 场景设定 / 分镜与镜头 / 生成实验 / 时间轴 Slot”。
- 内部 Agent 的业务讨论放 Agent Channel；明确的图片/视频生成放 Imagine Channel。
- 发起任务前先用现有 action 解析当前/目标 Channel，只有没有合适 Channel 时才创建。

## 内部 Agent 的使用策略

外部 Agent 只有自己的通用知识和操作 Syncast 的能力；项目内部有很多 Skill 只在 Syncast 内部 Agent 中可用，外部 Agent 无法直接访问。这些内部 Skill 覆盖：

- 如何使用项目内部模型和工具参数。
- 如何优化提示词。
- 剧本、角色、场景和镜头设计。
- 视频创作、镜头生成、媒体理解、项目规范模板等专业任务。

因此外部 Agent 应在需要创作判断时使用内部 Agent，范围限定为：

- 从零构思项目规范、剧本、角色设定、场景设定或风格规范。
- 不确定某个模型参数或工具参数的含义。
- 不确定图片/视频提示词应该如何组织。
- 需要拆分镜头、设计多镜头片段或整理时间轴 Slot。
- 需要内部 Skill 调用平台能力完成复杂业务任务。

以下任务不委托，直接通过项目 GraphQL 完成：用户已提供完整正文、精确字段更新、替换确定文本、清理指定段落，以及其它不需要创作判断的机械搬运。委托 goal 默认要求“只返回草案或创作判断，不执行项目写入”；外部 Agent 在用户确认的目标位置写入并做完整回读。不要把用户原文交给内部 Agent 再转述一遍。

内部执行器不是两套产品：项目里创建的每个 Agent 都是一份独立声明，可以直接作为顶层执行器，也可以被其它 Agent 绑定为命名选择。直接选择模型时，模型可发现项目全部 Agent；直接选择某个声明 Agent 时，它只能发现自己的绑定项。无论哪种顶层入口，都可以从头创建不继承父 Agent 指令的临时子 Agent；所有子执行均为叶节点，不会继续展开绑定关系。

外部 Agent 应显式传 `executor`。需要稳定的通用入口时使用 `{ kind: "model", model: ... }`；需要某个既有 Agent 的完整模型、指令和 Skill 声明时使用 `{ kind: "agent", agentId: ... }`。项目 Agent ID 与绑定关系从 Agents GraphQL 获取，不要把 CLI 外部身份的 `--agent-id` 当成内部 Agent ID。

声明 Agent 的 custom Skill binding 表达“明确选中”，`preload` 则独立表达“是否启动预载”：选中但 `preload=false` 的 Skill 仍可按需加载；`preload=true` 才会把完整说明放进启动上下文。全部内置 Skill 始终可按需加载，不需要用 binding 授权。`alwaysApply` custom Skill 始终可用并启动预载，Skill 自身的 `depends` 只表达依赖，不带 preload。

`allowLoadSkills` 是历史兼容字段，它现在只控制是否把**未选中的项目 custom Skill**也加入可发现范围。新 Agent 默认/建议显式写 `false`；旧 Agent 缺失该字段时按 `true` 保持历史行为。关闭它不会禁用已选 custom Skill、依赖、`alwaysApply` Skill或任何内置 Skill。

一次内部 Agent 任务使用发送时冻结的 Agent/Skill 快照。协作者在任务运行中修改配置不会让同一任务前后漂移，修改从下一次任务开始生效。root task 权限同时是所有命名/临时子 Agent 的上限；当前 UI 创建的项目 Agent 默认继承 root 权限。

默认 Agent 模型偏好 Opus 4.8；如果平台提供模型选择且用户没有特殊要求，优先使用它。注意控制上下文长度：不要把所有任务都放在同一个内部 Agent 对话里，按主题拆分 Channel 或任务，避免性能下降和上下文浪费。

## @ 引用与资源校验

Agent 输入、文档输入和 Imagine 输入都支持 `@` 引用项目内容。外部 Agent 应使用 action 获取真实 ID，再把引用写入输入中。

常用内部文本格式：

```text
@{asset:<assetId>}
@{doc:<docId>|<displayName>}
@{doc-section:<docId>:<sectionId>|<displayName>}
```

推荐流程：

1. 先用带 `query` 的 `syncast.assets.list`、带 `path` 的 `syncast.assets.browse` 或 `syncast.assets.get` 找到资产；需要交给外部 Agent 自行下载时，用 `syncast.assets.downloadUrls` 获取临时远端下载链接。
2. 记录资产名称、ID、文件夹和用途。
3. 在提示词或 references 参数中使用解析后的 `assetId`。
4. 发起生成前用 `syncast.assets.get` 确认每个引用 ID 仍存在；项目内生成继续使用项目 Action 和真实 Asset ID。
5. 只要有资产 ID 无法读取或同名搜索不唯一，停止生成并向用户说明，避免浪费积分。
6. 读与本次产物有关的项目规范，再浏览 Assets 目录树；规范已明确命名/分类时，记录目标名称与真实 folder ID。只在明确规范路径缺失时才调用 `ensureFolderPath`。

对于文档引用，先用 `syncast.docs.readForAgent` 获取文档索引和 outline。需要引用整个文档时用 `@{doc:...}`；只需要风格规范、场景规范等章节时，用 `@{doc-section:...}` 引用具体标题/章节。项目规范可以拆成多个子文档，也可以在一个文档中按标题组织，原则是方便人类阅读和后续单独引用。

为节省上下文，外部 Agent 可以在本地建立临时项目计划文件，缓存：

- 操作记录。
- 资产名称与 ID 对照。
- 文档 ID、章节 ID 和用途。
- 已使用的 Channel、任务 ref、生成结果位置。

临时文件只用于外部 Agent 速查；正式剧本、规范、角色、场景等项目内容应写入 Syncast 项目文档。

## 模型与生成策略

模型选择应保守，优先使用推荐模型：

- 图片模型：推荐 Nano Banana 2 和 OpenAI GPT Image 2；一般只使用 2K，质量使用 `auto`，这是默认最佳组合。
- 视频模型：只推荐 SeedDance 2.0；除非用户明确要求，否则使用 Fast 模式。SeedDance 2.0 / Fast 只允许 720P，禁止 1080P。
- 一般情况下不推荐其它模型，除非用户明确指定或内部 Agent 给出充分理由。

发起生成时应像人类一样选择合适模型和参数：

- 独立优化：调用 `syncast.imagine.optimizePrompt`。
- 真实生成：如果用户未说明项目内工作还是项目外独立资产，必须先询问。项目内未指定频道时使用 `syncast.imagine.submit`，指定真实现有频道时使用 `syncast.imagine.submitToChannel`；项目内禁止使用直接生成 CLI。只有用户明确选择项目外独立资产时才使用 `syncast imagine` / `video` / `audio` / `music` / `sound-effect`。

图片/视频生成前必须确认：

- 使用的模型和模式。
- 图片是否为 2K + `quality: auto`；SeedDance 2.0 / Fast 视频是否为 720P。
- 预计积分消耗和余额。
- 所有参考资产都已解析到正确 ID。
- 输出资产命名是否清晰，例如角色、场景、关键帧编号和镜头编号。
- 如果规范已确定归档目录，先在当前项目解析或创建目录，再把真实 `targetFolderId` 传给项目 Imagine Action；不要向项目 Action 猜名称或路径。
- 对分类确定的中间产物，提交生成时直接携带 `targetAssetName` + `targetFolderId`；只在需要根据实际画面/内容再分类时使用生成后移动。

## 项目规范与文档组织

项目刚开始时，外部 Agent 应先和用户讨论项目类型、目标和交付物。需要从零构思或结构未知时，委托内部 Agent 返回规范草案；用户已经给出完整内容或明确字段时，直接写入并回读，不再委托改写。

如果项目是电影感 3D 动画、三维渲染二维感、AI 动画短片或用户提到“3D 但不要塑料感”，项目规范的“风格规范 / 提示词规范”必须明确参考 `$cinematic-3d-animation`。委托内部 Agent 时，把这条写进 goal，要求其按该 Skill 的“用 3D 技术做 2D 审美”方法组织整体视觉原则、固定提示词骨架、正向词、反向控制和验收标准。

建议文档结构：

- 总体项目规范：项目目标、风格、世界观、视觉语言、禁忌项。
- 风格规范：整体风格提示词、镜头语言、色彩、质感、参考方向。
- 角色文档：每个角色的外观、服装、表情、动作、参考图和固定提示词。
- 场景文档：空间、时间、光线、道具、气氛和固定提示词。
- 分镜/剧本文档：剧情、镜头、运镜、转场、台词和生成计划。

常用项（如项目整体风格提示词）建议单独成文档或独立章节，方便后续通过 `@` 精确引用。

## 生成路线：资产参考与多镜头

具体生成建议分两种：

### 资产参考模式

先为角色、场景和关键物件生成或整理参考图。通过视频模型的全能参考模式生成时，`@` 这些资产，并用文本组织剧情、动作、镜头和情绪。

适合：

- 角色/场景一致性优先。
- 已有明确参考图。
- 要把多个固定资产放进同一画面或同一镜头。

### 多镜头合成生成

使用 SeedDance 2.0 视频模型时，可以把多个镜头合到一个 4 到 15 秒的生成片段中。为了更好的一致性和更自然的镜头切换，可以把多个不同运镜和转场写进同一个片段提示词。

适合：

- 希望一个请求覆盖一段连续剧情。
- 需要自然转场或连续动作。
- 多个镜头共享角色、场景和情绪。

## 协作创作的三种交付形式

当项目是外部 Agent 和用户一起创作时，应询问用户更偏好的形式：

- 项目文档形式：在文档里列剧情、分镜脚本、每个镜头提示词和所有参考。适合用户想先审稿。
- 时间轴 AI 占位 Slot：在时间轴上排布待生成镜头片段，让用户查看每个 Slot 并手动点击生成。适合用户想参与挑选和控制成本。
- 关键帧生成形式：复杂镜头或场景变化先生成关键帧，再决定是否做视频。适合需要精确控制空间、角度和角色一致性的片段。

生成关键帧时要合理命名，并确保每个关键帧关联的资产、场景空间位置和角度一致。例如使用“镜头 03 关键帧 01 / 02 / 03”这类名称，而不是泛泛的“生成图 1”。

## 模块地图

### 开始页

开始页是项目总览。适合接手项目后的第一站，用来确认项目名、简介、最近资产、最近画布、最近时间线和快捷入口。

Agent 判断：

- 项目是否刚创建。
- 是否缺项目简介。
- 是否已有资产、画布或时间线。
- 是否需要先补规范或整理素材。

### 资源

资源页是素材资产管理区，包括上传、文件夹、筛选、评分、收藏、导出和检查器。

Agent 判断：

- 有哪些图片、视频、音频和文件夹。
- 是否有可作为生成参考的资产。
- 是否需要先整理文件夹或解析资源名称。

用户未在当前请求中明确要求时确认；删除始终确认：

- 上传文件、新建/重命名/移动文件夹或资源。
- 删除文件夹或资源。
- 导出资产。

### 文档

文档页用于项目规范、剧本、角色设定、分镜、提示词规范等。项目缺规范时，通常应优先处理文档，而不是直接生成素材。

Agent 判断：

- 是否已有项目规范。
- 是否已有剧本、角色设定、场景设定或分镜说明。
- 是否需要内部 Agent 做创意构思，还是已有完整内容可直接写入。

用户已经明确要求的新建或编辑视为授权，不再重复确认。只有目标不明确、扩大范围、删除内容或需要额外积分时确认；会造成未请求内容丢失的整段覆盖按删除处理，用户明确要求的精确替换不重复确认：

- 删除文档或正文。
- 覆盖无法安全 patch 的既有正文。
- 应用会扩大内容范围或触发额外任务的模板。

### 画布

画布用于组织视觉素材、参考、分镜结构和创作草图。

Agent 判断：

- 是否已有页面。
- 是否有视觉结构、参考图或分镜布局。
- 是否可作为后续生成和内部 Agent 任务的上下文。

用户未在当前请求中明确要求时确认；删除始终确认：

- 新建页面。
- 添加、移动或编辑画布对象。
- 删除画布对象。
- 在画布侧边 Agent 发送消息。

### 时间轴

时间轴用于把素材按时间组织、预览、粗剪和导出。

Agent 判断：

- 项目是否进入剪辑阶段。
- 是否已有可排布素材。
- 是否需要先生成/上传素材再创建时间轴。

用户未在当前请求中明确要求时确认；删除始终确认：

- 创建时间轴。
- 修改时间线片段。
- 导出或渲染。

### AI / Agent

AI 页包含两类核心能力：

- Imagine：生成图片、视频、音频等内容。
- Agent：内部 Agent 对话，适合复杂业务判断、项目方案构思、创意性文档草案、创作规划和多步骤推理。

Agent 判断：

- 如果任务需要创意判断、项目方案、视频工作流或未知结构，用 `syncast.agent.delegate` 委托内部 Agent 返回草案；完整正文与精确修改直接走项目 GraphQL。
- 如果任务是明确生成图片/视频，先判断用户是在项目中工作还是只要项目外独立资产；不明确就先询问。项目内默认走 Action Bridge `syncast.imagine.submit`，只有指定真实现有频道才用 `submitToChannel`，不得回退到直接 CLI。
- 生成前先调用 `syncast.billing.summary` 或 `syncast.imagine.estimateCredits` 告知余额和预计消耗。

用户没有明确要求相应计费动作或预算时确认：

- 发送内部 Agent 消息。
- 发起图片、视频、音乐或其它生成任务。
- 运行会消耗积分的内置技能。

### 工作流

工作流用于节点式自动化编排，适合重复创作流程，例如批量生成分镜、批量处理素材、生成摘要或串联 LLM 节点。

用户未在当前请求中明确要求时确认；运行产生额外积分消耗时单独按预算确认：

- 修改节点、边、参数和描述。
- 运行工作流。

### 录制与绘画

录制用于视频、照片、音频采集；绘画用于画板、图层和 AI 辅助编辑。

即使用户已提出方向，也要在启用设备能力或产生计费前确认具体范围：

- 打开摄像头、麦克风或屏幕共享。
- 开始录制。
- 保存绘画结果。
- 使用消除背景、高清增强、智能分层等 AI 辅助。

## 默认接手流程

当用户没有给出非常明确的单步操作时，外部 Agent 应按以下顺序接手项目：

1. 初始化外部 Agent 身份。
2. 调用 `syncast.project.inspect` 获取项目概要。
3. 判断是空项目还是已有操作项目。
4. 判断或询问用户选择“协作创作”还是“完全托管”。
5. 检查同步、任务、积分、文档、资源、频道、时间轴和模型概要。
6. 规划少量稳定 Channel，不要每个任务都新建 Channel。
7. 如果项目内容不足，向用户汇报缺口：缺规范、缺素材、缺画布结构、缺时间轴或缺任务结果。
8. 如果需要创意构思或未知结构，用 `syncast.agent.delegate` 获取草案；如果用户已给出完整内容或精确修改，直接使用项目 GraphQL。
9. 如果需要排布待生成镜头，优先创建时间轴 AI Slot；协作模式让用户先检查，明确授权代操作时可用 `syncast.timeline.generationSlots.submit` 触发。
10. 如果需要生成素材，先读相关规范和当前 Assets 目录，解析并校验引用、确认目标命名/目录、估算积分；未指定频道时用 `syncast.imagine.submit`，指定现有频道时用 `syncast.imagine.submitToChannel`，文档块或时间轴 Slot 使用专用 Action；项目内始终禁止直接生成 CLI。需要时先单独调用 `syncast.imagine.optimizePrompt`。
11. 等待长任务完成：`window.__syncastAgent.wait(ref, { returnResult: true })`。
12. 读取结果并复查：文档用 `syncast.docs.readForAgent`，资源用 `syncast.assets.browse/list/get`，频道用 `syncast.channel.messages.list/get`；时间轴 Slot 用 `syncast.timeline.get`，最终成片或轨道结构用 `syncast.timeline.context.load`，已有素材理解映射用 `syncast.timeline.understanding.load`，按内容检索时间轴片段用 `syncast.timeline.semantic.search`，缓存维护状态用 `syncast.assets.cacheLifecycle.inspect`。
13. 向用户汇报完成内容、产物位置、风险和下一步建议。

## 风险边界

可以直接执行的低风险动作：

- 读取项目概要、文档索引、资源摘要、频道摘要、任务状态和通知。
- 查看可用模型与积分余额。
- 等待已有任务完成并读取结果。
- 校验资产/文档引用是否存在。

用户在当前请求中明确要求的同范围、非计费项目写入视为已授权，不要在新建文档、编辑正文、创建 Skill 或精确字段更新前重复询问。

只有以下情况需要再次询问：

- 超出用户原始要求的范围扩张。
- 任何额外消耗积分的生成、内部 Agent 消息或工作流运行；用户已明确授权该计费动作时不重复确认。
- 删除内容；会造成未请求内容丢失的整段覆盖按删除处理，用户明确要求的精确替换不重复确认。
- 目标文档、Skill、字段、项目或执行方式不明确。
- 摄像头、麦克风、屏幕共享和录制。
- 同步管理器中的强制重载、切通道、关闭项目 tab 等恢复/协作操作。

建议询问方式：

```text
我已经看完当前项目状态。你要求的文档写入可以直接执行，不需要再次确认；其中生成/内部 Agent 任务会额外消耗积分，请确认是否也执行这部分，并告诉我允许使用的模型或预算。
```

## 汇报格式

当用户只要求“了解项目”或“看看现在是什么状态”时，推荐汇报：

- 项目是否为空。
- 文档、资源、画布、时间轴分别有什么。
- 是否有正在运行或失败的任务。
- 同步是否正常。
- 当前积分/团队扣费策略是否足够支持下一步。
- 可直接继续的低风险动作。
- 需要用户确认的高风险动作。

示例：

```text
我看完了。这个项目目前缺少资源和时间轴，已有一份项目规范文档和一个画布页面；任务列表没有运行中任务，同步状态正常。下一步可以先让内部 Agent 基于规范生成分镜方案，或先上传参考素材。如果要生成图片/视频，需要你确认模型和预算，因为会消耗积分。
```
